A.2
2026 modernization deltas vs 2022 spec assumptions
DECISIONS reference
OS minimums (must renegotiate)
  • iOS 9.3 (2016) → iOS 14+ (covers iPhone 6s and newer, ~99% of active devices in 2026). Modern Expo requires iOS 13.4+.
  • iPadOS 10 (2016) → iPadOS 14+ (same rationale).
  • Android 5.0 (2014) → Android 8+ (API 26, covers ~95% of active devices). Modern Expo requires API 23+, Play Store target API 35 in 2026.
  • Android TV 9.0 → kept (already reasonable).
  • Android STB 5.0 → Android 7+ ideally (depends on RTSH-issued hardware — confirm in P0.12).
  • Tizen 2.3 (2015) → Tizen 5.0+ (2019+). Tizen 2.3 era ships ES5-only engines that won't run modern React without aggressive transpilation.
  • WebOS 2.0 (2014) → WebOS 4.0+ (2018+). WebOS 6+ uses Chromium and is dramatically easier to target.
Stack picks: 2022 default → 2026 first-party
  • React Native <0.70 + manual ABI builds → Expo SDK 55 + RN 0.83 with EAS Build cloud CI
  • react-native-video → expo-video (Expo first-party, GA SDK 51, built on AVPlayer + ExoPlayer; supports HLS + AES-128 + Widevine/PlayReady/ClearKey via drmOptions). Fallback to react-native-video if header injection on AES-128 key requests is insufficient.
  • react-native-track-player → expo-audio (SDK 54+, lock-screen controls via setActiveForLockScreen, background audio config plugin).
  • react-native-orientation-locker → expo-screen-orientation (Expo first-party).
  • react-native-device-info → expo-device (Expo first-party).
  • React Native legacy → New Architecture (Fabric + TurboModules + Bridgeless) enabled by default in modern Expo. Hermes ON.
  • Manual JS hotfix delivery → EAS Update for JS-only OTA hotfixes (no store review needed for JS changes).
  • Android TV with separate bare RN repo → Expo + react-native-tvos as the RN dependency (officially supported by Expo since SDK 54). Same Expo workflow.
  • No analytics opinion → Sentry for crash/errors + standardized event interface in packages/analytics with swappable provider (Mixpanel / PostHog / self-hosted).
  • moment.js (if implied) → date-fns or luxon (smaller bundle, no global mutation).
  • Class components / context spaghetti → react-hook-form + zod + zustand + TanStack Query (the modern composable stack).
DRM: spec mandates AES-128 — modern context

AES-128 HLS encryption is the lightest DRM tier — handled natively by every mainstream HLS player including expo-video. Acceptable for public broadcast (RTSH's case) but NOT studio-grade. Determined attackers can intercept the key over HTTPS and decrypt. For premium content (movies, paid sports), 2026 industry standard is Widevine L1/L3 (Android) + FairPlay (Apple) + PlayReady (Tizen/WebOS). Roadmap supports AES-128 today + leaves the player abstraction in packages/player so we can upgrade to Widevine/FairPlay later without rewriting screens.

Store policy changes since 2022
  • Google Play 14-day testing rule (effective 2024 for new orgs): 12 closed testers × 14 days before production launch. Plan calendar accordingly (P0.9, P6.4).
  • Apple App Tracking Transparency (ATT) + Privacy Manifests: required for any cross-app tracking. With anonymous IP-based geo and no fine-location, we likely sidestep ATT prompts — confirm during P5.1 GDPR audit.
  • Android target API 35 (Android 15) required for new apps by mid-2026. Expo SDK 55 covers this automatically.
  • Samsung Tizen Game Hub vs TV Apps: distinct review tracks since 2024 — we submit to TV Apps.
  • Apple Sign in with Apple mandate: still applies if we add any third-party social login later.

B.1 Encoding pipeline

Convert broadcast signals into ABR HLS streams.
  • SDI + TSoIP (UDP MPTS) signal ingest from RTSH studio
  • Primary + backup encoder farm with <1min failover (spec Enc.5)
  • ABR ladders per channel: HD 1080/720/576/360, SD 2×576+360, Radio audio-only
  • Codecs: H.264 + H.265 video, AAC / HE-AAC / MP3 / AC3 / MPGA audio
  • Per-channel bitrate config editable by RTSH staff
  • Output: HLS playlists + segmented chunks to origin
Owner: broadcast engineering team · Spec ref: Shtojca 5 §2 (12 items)

B.2 Middleware (auth, entitlement, geoblock, ads, CMS)

Business logic layer + admin tools for RTSH ops staff.

App-facing APIs:

  • POST /auth/register, POST /auth/login, POST /auth/refresh, POST /auth/forgot-password, POST /auth/reset
  • GET /channels — ordered list with logos, packages, per-device stream URLs
  • GET /epg?channel=&day= — events with rating, title, start, end
  • GET /catchup?eventId= — VOD URL (HLS, AES-128)
  • GET /stream/{channelId}?device=mobile|tv|stb — device-class-specific manifest URL
  • GET /theme — colors, logos, banners (server-driven UI)
  • GET /config — catchupDays, ad rules, T&C version, parental defaults
  • GET /ads?channel=&at=ISOTimestamp — active ad for channel + time slot
  • POST /analytics — event ingestion endpoint
  • GET /legal/{terms|privacy}?version= — versioned legal text

Admin CMS (RTSH-facing): user management, geoblocking per channel/event GUI, channel reorder, ad scheduling, theme tokens, T&C editor.

Security: JWT issuance, refresh-token rotation, AES-128 key endpoint (HTTPS, token-gated, rate-limited).

Owner: backend team · Spec ref: Shtojca 5 §3 (14 items)

B.3 Stream distribution / CDN

Origin → edges → end users globally.
  • Origin server generates m3u8 + chunks
  • Geo-distributed edge fleet (EU + US per spec Distr.16)
  • AES-128 key delivery endpoint (token-gated, rate-limited, periodic rotation)
  • Adaptive bitrate routing — edge serves closest user
  • IP + domain blocking
  • DDoS protection at origin + edge
  • Auto-scaling on traffic spikes
  • Edge-middleware integration for geoblocking (token validation includes geo-region)
Owner: infra / DevOps team · Spec ref: Shtojca 5 §4 (16 items)

B.4 EPG ingestion

Pull XMLTV from RTSH FTP, parse, expose JSON.
  • Scheduled FTP poll for new XMLTV files
  • Parse + normalize to consistent JSON (UTC timestamps, age ratings, categories)
  • Store with channel + day index
  • Expose via middleware GET /epg endpoint
  • Validation: reject malformed XMLTV before publishing (it would break the entire UI)
Owner: backend team · Spec ref: Shtojca 5 §1.16

B.5 Catch-up recording

Continuous channel recording → VOD packaging.
  • Continuous recording per channel (24/7)
  • Rolling 7-day retention (configurable per Spec App.18)
  • Generate per-event HLS VOD playlists keyed by EPG event ID
  • Same AES-128 DRM as live streams
  • Same ABR ladders as live
Owner: backend + storage team · Spec ref: Shtojca 5 §1.17–19

B.6 Central monitoring (NOC)

24/7 operational dashboard for RTSH operators.
  • Multiviewer: every channel preview in one screen
  • Alarm log retention: 1 year minimum
  • Health checks: audio + video presence per channel, bitrate, origin status
  • Real-time user stats: count, geolocation, audience by channel, platforms (mobile/TV/STB)
  • CDN traffic reports: bandwidth per edge, traffic over time
  • IP-gated dashboard URL (only accessible from RTSH IP range)
  • Always-on (24h × 365 days)
Owner: NOC / operations team · Spec ref: Shtojca 5 §5 (8 items)
